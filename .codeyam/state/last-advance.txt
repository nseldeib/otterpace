Checklist:
[ ] Recapture screenshots: `codeyam-editor-dev editor recapture-stale --skip-when-clean --include-aged --max-seconds 300 --max-attempts 3`
    Exits non-zero if (a) any per-slug capture failed, (b) the dev server is
    unhealthy on pre-flight, or (c) the overall budget expired with zero
    successes. If the pre-flight bails with "dev server unhealthy", fix the
    dev server first — do NOT pass `--skip-when-clean` to bypass it (the flag
    only short-circuits the empty-coverage fast path, not the pre-flight).
    `stage-feature` will also refuse to run if coverage still shows stale or
    missing screenshots, so dismissing per-slug failures is no longer possible.
    `--skip-when-clean` short-circuits with `{"skipped_clean": true}` when
    audit already reported zero stale + zero missing — common case for
    backend-only features. `--include-aged` unions the audit's
    `AGED_STALE_SCREENSHOTS` set into the recapture target, so a scenario
    whose source mtime crossed the freshness threshold (e.g. an unrelated
    `git checkout` bumped it forward) is re-captured here — without it,
    the dep-tree-only check would report `0 stale` while the audit gate
    afterward still fires on those slugs, leaving the workflow stuck.
    `--max-seconds` caps each attempt at 5 minutes;
    individual slugs have their own 60s timeout. `--max-attempts 3` retries
    after a per-attempt budget hit (stdout JSON `budget_expired: true`) so a
    single 300s window can't silently leave stale screenshots behind — each
    retry re-evaluates coverage and only touches slugs that are still stale.
    The final stdout JSON includes `attempts`, `budget_expired`, and
    `skipped_count` so you can confirm the run drained the set. Do NOT pipe
    this through `tail` — full output (per-slug `→ Capturing` / `✓` / `✗` /
    `⏱` lines plus the `▶ Pass N/M` startup line) is the only signal you
    have if a scenario hangs. Per-slug timeouts now fail the command;
    investigate the dev server or the scenario before retrying.
[ ] Stage the feature: `codeyam-editor-dev editor stage-feature \
       --path <feature-file-1> --path <feature-file-2> ...`
    Pass --path for every NON-`.codeyam/` file you edited during this
    feature. All non-ignored `.codeyam/` files are staged automatically
    — do NOT ask the user about any `.codeyam/` path, and do NOT pass
    them via --path. `.gitignore` is the authoritative skip-list.
    If you have many changed files, you can use `stage-feature --from-status`
    to stage every dirty file in the working tree. Use `--exclude-prefix`
    to skip directories you didn't touch.
    `stage-feature` accepts both file paths and directory paths. File paths
    work even if they are currently untracked inside an untracked directory.
    The command prints `stagedInternal` (auto), `stagedExternal`
    (your --path args), and `untouchedExternal` (other dirty files not
    yours — leave them for the `push` slug). If `untouchedExternal` is
    non-empty, that is fine; the `push` slug's `commit-state` will surface them.
    `stage-feature` also runs `session-cleanup` (orphan screenshots, log
    rotation, tmp prune) per `editor.json::cleanup` BEFORE the dirty-tree
    scan, so any deletions land in `stagedInternal` and fold into the
    feature commit. Best-effort; failures log to stderr.
[ ] If `.gitignore` was modified, untrack newly-ignored files:
    `git ls-files --cached | git check-ignore --no-index --stdin`,
    then `git rm --cached <file>` for each match and re-run
    `stage-feature`.
[ ] Commit: `git commit -m "feat: <title>" -m "<journal description>"`
    The commit message body MUST match the journal description exactly.
    Verify: git rev-parse HEAD

[ ] If the pre-commit hook fails, read the error, fix the issue,
    re-run `stage-feature` (the cache regeneration may dirty more
    `.codeyam/` files — they get auto-staged), and commit again.
[ ] Review the commit with `git log --stat -1` to confirm the right files were included
[ ] Don't worry about regenerated .codeyam/ caches dirtied AFTER this commit
    (post-commit hook output) — the `push` slug captures any such leftovers in a
    separate `chore: update codeyam state` commit before the push. Caches
    already dirty BEFORE the commit are now folded into the feature commit
    itself by `stage-feature`.
